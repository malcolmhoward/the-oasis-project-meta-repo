# PFT Workflow: Generating Foundation Files for O.A.S.I.S. Components

This guide documents how to use the [Project Foundation Template (PFT)](https://github.com/malcolmhoward/project-foundation-template) to generate governance files for O.A.S.I.S. component repos.

PFT is included as a submodule at `repos/project-foundation-template/`. Always invoke this copy — not a separate clone or installed version — to ensure version consistency across the ecosystem.

---

## Quick Reference

```bash
# From the meta-repo root:
cd repos/project-foundation-template

python generate_foundation.py \
  --preset minimal \
  --project-name "ComponentName" \
  --author-name "The O.A.S.I.S. Project" \
  --output-dir /tmp/pft-output

# Then validate before committing:
python validate_customization.py /tmp/pft-output --verbose
```

Replace `ComponentName` with the component's display name (e.g., `"MIRAGE"`, `"S.T.A.T."`, `"O.A.S.I.S. Simulation Framework"`).

## Step-by-Step Workflow

### 1. Navigate to the PFT submodule

```bash
cd repos/project-foundation-template
```

Always use this copy. It is pinned to the ecosystem-approved version (`feat/v3.7.0-generation-log` branch).

### 2. Generate foundation files

```bash
python generate_foundation.py \
  --preset minimal \
  --project-name "ComponentName" \
  --author-name "The O.A.S.I.S. Project" \
  --output-dir /tmp/pft-output
```

**Why these parameters:**

| Parameter | Value | Reason |
|-----------|-------|--------|
| `--preset` | `minimal` | Appropriate for all O.A.S.I.S. components; covers core governance without over-prescribing |
| `--author-name` | `"The O.A.S.I.S. Project"` | Ecosystem-level attribution; avoids individual contributor names in generated boilerplate |
| `--output-dir` | A temp directory | Lets you review generated files before merging into the component repo |

PFT will prompt you to acknowledge the ethical use agreement before generating files. Read it — it's intentional.

### 3. Review generated files

```bash
ls /tmp/pft-output/
# CONTRIBUTING.md  FOUNDATION_NOTICE.md  LICENSE  README.md
```

Compare each generated file against what the component repo already has:

| File | If it doesn't exist | If it already exists |
|------|--------------------|-----------------------|
| `README.md` | Copy, then customize | Keep existing; extract relevant sections from PFT output |
| `CONTRIBUTING.md` | Copy, then customize with O.A.S.I.S. templates (see below) | Merge — prioritize O.A.S.I.S.-specific fork workflow |
| `LICENSE` | Compare with existing O.A.S.I.S. component licenses first — all use GPL-3.0; use GPL-3.0 to align with the ecosystem | Compare — likely identical; keep existing if it already exists |
| `FOUNDATION_NOTICE.md` | Copy as-is | Copy as-is |

### 4. Use O.A.S.I.S. templates for customization

Pre-customized O.A.S.I.S. templates live in `templates/oasis-foundation/`:

```bash
ls templates/oasis-foundation/
# CLAUDE-template.md  CONTRIBUTING-template.md  README-template.md
```

These are starting points with O.A.S.I.S.-specific sections (MQTT topics, hardware requirements, ecosystem links) already filled in. Use them alongside PFT output — don't start customization from scratch.

### 5. Validate before committing

```bash
cd repos/project-foundation-template
python validate_customization.py /tmp/pft-output --verbose
```

This checks that:
- Placeholder values (e.g., `[PROJECT NAME]`, `[YOUR NAME]`) have been replaced
- Required sections are present
- Files don't contain uncustomized boilerplate

Fix any warnings before copying files into the component repo. For CI enforcement:

```bash
python validate_customization.py --strict ./path/to/component
```

### 6. Copy to component repo and commit

```bash
# Copy the files you want into the component repo
cp /tmp/pft-output/CONTRIBUTING.md /path/to/component/
cp /tmp/pft-output/LICENSE /path/to/component/

# Create CLAUDE.md from the template (not generated by PFT — write it fresh)
cp templates/oasis-foundation/CLAUDE-template.md /path/to/component/CLAUDE.md
```

Commit with a conventional commit message:

```bash
git commit -m "docs(governance): Add PFT foundation files

Generated with PFT v3.7.0 (minimal preset), customized for <ComponentName>.
Follows O.A.S.I.S. ecosystem standard (ADR-0001).

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>"
```

## Preset Selection

For O.A.S.I.S. components, use `--preset minimal` unless there is a specific reason to go broader.

| Preset | Principles | When to use |
|--------|------------|-------------|
| `minimal` | 3 | All O.A.S.I.S. component repos (current standard) |
| `light` | 6 | If the component has active external contributors |
| `standard` | 9 | If the component handles sensitive data or security concerns |
| `strict` | 12 | Not currently used in the ecosystem |

**Why minimal?** O.A.S.I.S. components are primarily coordinated through S.C.O.P.E. Governance beyond minimal creates noise without benefit at the current project scale. Upgrade preset as the ecosystem matures.

## Adopting PFT in an Existing Repo

If the component repo already has `README.md`, `CONTRIBUTING.md`, or `CLAUDE.md`:

1. **Do not overwrite blindly.** Generate to a temp directory first.
2. **Merge rather than replace.** Keep the existing README's content; add PFT's structure around it.
3. **CONTRIBUTING.md priority order:** O.A.S.I.S. fork-first workflow → PFT ethical guidelines → component-specific details.
4. **CLAUDE.md is always written fresh** from `templates/oasis-foundation/CLAUDE-template.md` — it is not generated by PFT.
5. **LICENSE:** All O.A.S.I.S. components use GPL-3.0, not PFT's default MIT. If the component already has a license file, keep it. If it doesn't, use GPL-3.0 to align with the ecosystem — do not copy PFT's MIT template as-is.

## Troubleshooting

**"VERSION OUTDATED" warning on generation:**
PFT has an advisory expiration date. Read and acknowledge the warning — it is intentional (encourages keeping governance practices current). Type `I understand the risks` at the prompt, then `yes` to the ethical agreement.

**"EOF when reading a line" error:**
PFT requires interactive input for ethical agreement prompts. Pipe responses when running non-interactively:

```bash
(printf "I understand the risks\nyes\nyes\nyes\n") | python generate_foundation.py ...
```

**Generated README overwrites project-specific content:**
Generate to `/tmp/` and manually merge. The PFT README is a template — it does not know your component's architecture. Keep your existing README as the primary; extract relevant governance notices from PFT's output.
